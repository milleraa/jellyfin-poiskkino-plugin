# Роли и ответственность

| Роль | Ответственность |
|------|-----------------|
| **Стейхолдер** | Продуктовые решения, приёмка PR/MR, доступы |
| **Оркестратор** | Процесс, `status.md`, bootstrap, эскалации |
| **Аналитик** | Требования и AC для **новых фич** |
| **Архитектор** | Дизайн системы, границы, ADR |
| **Техлид** | Декомпозиция, задачи разработчикам, приёмка до QA |
| **Разработчик** | Реализация по задаче и стеку |
| **DevOps** | Docker, CI/CD, deploy scripts, Helm/Kubernetes, troubleshooting поставки |
| **QA** | Тест-дизайн, прогоны, Playwright, вердикт |
| **Финализатор** | DoD, доки, PR/MR delivery, архив после acceptance |

Промпты ролей: [`.cursor/agents/`](../../.cursor/agents/).

## Взаимодействие

- Все роли кроме Оркестратора вызываются как отдельные подзадачи/субагенты с ролевыми промптами из [`.cursor/agents/`](../../.cursor/agents/).
- Оркестратор делегирует одну роль за раз, ждёт результат, фиксирует handoff в `status.md` и только после этого запускает следующую роль.
- Аналитик ↔ Стейхолдер — уточнение «что».
- Аналитик → Оркестратор → Стейхолдер — обязательный gate после аналитики новой фичи: продолжать к Архитектору или оставить в backlog.
- Архитектор ↔ Аналитик — уточнение для дизайна.
- Техлид ↔ Разработчики/DevOps — задачи и ревью.
- Техлид → DevOps — обязательный impact check после разработки перед QA.
- DevOps ↔ Оркестратор/Стейхолдер — доступы, runners, секреты и окружения.
- QA ↔ Техлид — дефекты и доработка.
- Финализатор ↔ Оркестратор — PR/MR delivery, возврат rejected PR/MR, архив после acceptance.
