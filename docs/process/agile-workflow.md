# Agile workflow

## Типы работ

| Тип | Аналитик | Далее |
|-----|----------|--------|
| Новая фича | Обязателен | Аналитик → **Стейхолдер решает продолжать или оставить в backlog** → Архитектор → Техлид → Разработчики/DevOps → … |
| Bugfix с ясной постановкой | Опционален | Оркестратор → Архитектор (или Техлид при тривиальности) → Разработчики/DevOps → … |
| Hotfix | Как bugfix | Папка `docs/backlog/hotfixes/<name>/`, ускоренный поток в `status.md` |

## Цепочка

1. **Оркестратор** — intake, классификация, bootstrap пустого проекта.
2. **Оркестратор** — создаёт ветку и единственный `git worktree` для feature/hotfix, фиксирует их в `status.md`.
3. **Аналитик** (фичи) — `analysis.md`, вопросы Стейхолдеру.
4. **Оркестратор → Стейхолдер** — обязательная остановка `analysis-review`: продолжать сейчас или оставить фичу в backlog (`backlog-paused`).
5. **Архитектор** — только после подтверждения продолжения; `architecture.md`, ADR при необходимости; может вернуть к Аналитику.
6. **Техлид** — `tasks/*.md`, последовательное назначение разработчиков/DevOps, приёмка результата.
7. **Разработчик** (стек) — реализация по задаче.
8. **DevOps impact check** — после разработки Техлид проверяет, нужны ли DevOps-изменения из-за новых env vars, интеграций, портов, миграций, CI/deploy/infra последствий.
9. **DevOps** — Docker, CI/CD, deployment/infra задачи, если Оркестратор/Техлид решили подключить роль или impact check нашёл необходимость.
10. **QA** — после приёмки Техлидом и закрытого DevOps-check; Playwright по стратегии.
11. **Финализатор** — push, PR/MR, commit/push ссылки на PR/MR, удаление локального worktree, стадия `pr-mr-ready`.
12. **Стейхолдер** — review PR/MR: принятие/merge или возврат в `rework`.
13. **Финализатор** — после принятия/merge переносит работу в [archive](../archive/README.md) и обновляет историю.

Каждая роль после своего шага делает локальный commit без push, если меняла файлы. Push и PR/MR выполняет только Финализатор.

После `analysis-review` и подтверждения продолжения happy path идёт без плановых остановок до `pr-mr-ready`. Новые остановки допустимы только при блокере, эскалации к Стейхолдеру или rejected PR/MR.

## Делегирование через подзадачи

Оркестратор не исполняет все роли одним агентом. Каждый переход к роли означает отдельный вызов подзадачи/субагента с соответствующим ролевым промптом из `.cursor/agents/`.

Порядок на каждом шаге:

1. Оркестратор выбирает следующую роль по текущей стадии и `status.md`.
2. Оркестратор запускает подзадачу/субагента, передав feature/hotfix папку, `status.md`, релевантные артефакты и конкретный ожидаемый результат.
3. Роль выполняет свой шаг, обновляет артефакты, делает локальный commit без push, если меняла файлы, и возвращает отчёт.
4. Оркестратор принимает результат, обновляет handoff в `status.md` и только затем запускает следующую роль или эскалирует блокер Стейхолдеру.
5. Исключение: после Аналитика по новой фиче Оркестратор всегда останавливается на `analysis-review` и ждёт решения Стейхолдера перед Архитектором.

## Возврат PR/MR в работу

`pr-mr-ready` не означает `accepted`. Если Стейхолдер не принимает PR/MR, Оркестратор применяет `.cursor/skills/pr-mr-rework/SKILL.md`, сохраняет feedback в `status.md`, восстанавливает worktree из существующей feature/hotfix ветки и возвращает работу на нужную роль.

Цикл: `pr-mr-ready` → `rework` → нужная роль → `finalization` → `pr-mr-ready`. В архив переносится только принятая или merged работа.

## Где смотреть статус

В папке фичи/hotfix: **`status.md`** — текущая стадия и owner.
