# Branching (GitFlow)

## Ветки по умолчанию

| Ветка | Назначение |
|-------|------------|
| `main` | Продакшен-ready код |
| `develop` | Интеграция фич |
| `feature/*` | Новая функциональность |
| `fix/*` | Исправления в develop |
| `release/*` | Подготовка релиза |
| `hotfix/*` | Срочный фикс от `main` |

Имена `main`/`develop` могут отличаться — зафиксируйте в [git-remotes](../environments/git-remotes.md).

## Правила

- В защищённые ветки только через **PR/MR**.
- На одну feature/hotfix создаётся ровно один `git worktree` и одна рабочая ветка.
- **Оркестратор** создаёт ветку и worktree после intake/classification и фиксирует их в `status.md`.
- Задачи выполняются строго последовательно в рамках одного worktree; task-branches и task-worktrees не используются.
- Каждая роль после своего шага делает локальный commit без push, если меняла файлы.
- Коммиты: императив, при необходимости conventional commits; отдельные задачи и ролевые шаги фиксируются отдельными commit.
- Обычные роли не выполняют `git push`. Push, создание PR/MR и повторный push ссылки на PR/MR выполняет только **Финализатор**.
- После создания PR/MR Финализатор записывает ссылку в `status.md` / `finalization.md`, делает отдельный commit с этой ссылкой, выполняет повторный push и только затем удаляет локальный worktree.
- Финализатор удаляет только локальный worktree. Рабочая ветка и remote branch остаются до merge/close PR/MR.

Подробнее: `.cursor/rules/20-gitflow.mdc`.
